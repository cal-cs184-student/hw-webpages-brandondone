Part 1: Ray Generation and Scene Intersection

Question: Walk through the ray generation and primitive intersection parts of the rendering pipeline.

Overview

The pipeline starts in pathtracer.cpp. PathTracer::raytrace_pixel(x, y) treats (x, y) as the bottom-left corner of a pixel, draws ns_aa random 2D offsets from gridSampler, and converts each jittered sample into normalized image coordinates by dividing by sampleBuffer.w and sampleBuffer.h. For each sample, it calls camera->generate_ray(...), traces that ray with est_radiance_global_illumination(...), averages the returned radiance, and writes the final color into the sample buffer.

Ray generation happens in camera.cpp. Camera::generate_ray(x, y) maps normalized image coordinates from [0,1] x [0,1] onto the virtual sensor plane at z = -1 in camera space. The sensor extents are set by tan(hFov / 2) and tan(vFov / 2), so (0,0) lands at the bottom-left of the sensor and (1,1) lands at the top-right. That camera-space direction (sensor_x, sensor_y, -1) is rotated into world space with c2w, normalized, and packaged into a Ray whose origin is pos. The ray's valid interval is initialized to [nClip, fClip], so anything before the near plane or beyond the far plane is ignored immediately.

Intersection Flow

Tracing enters pathtracer.cpp in est_radiance_global_illumination. Right now it asks bvh->intersect(r, &isect) for the nearest hit. In this codebase, bvh.cpp is still the starter implementation, so it effectively loops over all primitives and calls each primitive's intersect(...).

The key contract is that every primitive must only report hits with t inside r.min_t and r.max_t, and if it finds one, it must shrink r.max_t to that closer hit. That is how the nearest intersection wins without extra sorting.

The hit record is stored in intersection.h. A valid primitive intersection fills in isect.t, isect.n, isect.primitive, and isect.bsdf. After that, the current starter shading code returns either a debug color from ray direction or a normal-based color from the surface normal.

Triangle and Sphere Intersection

Triangle intersection is implemented in triangle.cpp. Both has_intersection(...) and intersect(...) use a Moller-Trumbore-style solve. They compute barycentric coordinates and reject any hit outside the triangle or outside the ray interval. On a valid hit they set r.max_t = t. The full intersect(...) version also interpolates the shading normal from n1, n2, and n3 using barycentric weights, normalizes it, and stores the primitive and BSDF.

Sphere intersection is implemented in sphere.cpp. Sphere::test(...) solves the quadratic for the two ray-sphere roots. has_intersection(...) and intersect(...) pick the nearest root that lies within [r.min_t, r.max_t], then tighten r.max_t. The full intersect(...) computes the hit point r.o + t * r.d, uses the analytic sphere normal (hit_p - center).unit(), and fills the same intersection fields as the triangle case.

End-to-End Path

From pixel sample to shading result, the flow is:

  1. Sample a point inside the pixel in pathtracer.cpp.
  2. Generate a world-space ray in camera.cpp.
  3. Traverse the scene through pathtracer.cpp and bvh.cpp.
  4. Find the nearest primitive hit in triangle.cpp or sphere.cpp.
  5. Return the shading result to the pixel accumulator.
