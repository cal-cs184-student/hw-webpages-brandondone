Overview

In this assignment, I built a full Monte Carlo path tracer from the camera ray stage all the way to global illumination and adaptive sampling. I first implemented ray generation and primitive intersection for triangles and spheres, then added BVH acceleration to make scene traversal efficient. After that, I implemented two direct-lighting estimators, one using uniform hemisphere sampling and one using light importance sampling, and extended the renderer to handle indirect illumination through recursive path tracing with Russian Roulette. Finally, I implemented adaptive sampling so each pixel can stop early or continue sampling based on its estimated variance.

The most interesting part of this homework was seeing how rendering quality depends on both physical modeling and computational strategy. Better light transport models, like indirect bounces, make images much more realistic through effects such as soft shadows and color bleeding, but they also introduce significant noise and cost. Techniques like BVH acceleration, importance sampling, Russian Roulette, and adaptive sampling show how much of rendering is really about making those expensive light calculations practical. This homework made it clear how geometry, probability, and performance all come together in a path tracer.

Part 1: Ray Generation and Scene Intersection

  1. Walk through the ray generation and primitive intersection parts of the rendering pipeline.

    The pipeline starts in pathtracer.cpp. PathTracer::raytrace_pixel(x, y) treats (x, y) as the bottom-left corner of a pixel, draws ns_aa random 2D offsets from gridSampler, converts each jittered sample into normalized image coordinates using sampleBuffer.w and sampleBuffer.h, and then calls camera->generate_ray(...). Each generated ray is traced with est_radiance_global_illumination(...), the returned radiance values are averaged, and the final result is written into the sample buffer.

    Ray generation happens in camera.cpp. Camera::generate_ray(x, y) maps normalized image coordinates from [0,1] x [0,1] onto the virtual sensor plane at z = -1 in camera space. The sensor extents come from tan(hFov / 2) and tan(vFov / 2), so (0,0) lands at the bottom-left of the sensor and (1,1) lands at the top-right. The camera-space direction (sensor_x, sensor_y, -1) is rotated into world space with c2w, normalized, and packaged into a Ray whose origin is pos. The valid interval is initialized to [nClip, fClip], so any hit before the near plane or beyond the far plane is ignored.

    Tracing continues in est_radiance_global_illumination, which asks bvh->intersect(r, &isect) for the nearest hit. In the starter code, bvh.cpp still effectively loops over every primitive and calls that primitive's intersect(...) method. The important rule is that each primitive only accepts hits within r.min_t and r.max_t; if it finds a valid closer hit, it updates r.max_t. That is what keeps the nearest intersection so far.

    The final hit data is stored in intersection.h. A successful primitive intersection fills in isect.t, isect.n, isect.primitive, and isect.bsdf. From there, the current starter shading code returns either a debug color from ray direction or a color based on the surface normal.

    Triangle intersection is implemented in triangle.cpp using a Moller-Trumbore-style solve. Both has_intersection(...) and intersect(...) compute barycentric coordinates, reject hits outside the triangle or outside the ray interval, and set r.max_t = t when a valid hit is found. The full intersect(...) version also interpolates the shading normal from n1, n2, and n3 and stores the primitive and BSDF.

    Sphere intersection is implemented in sphere.cpp. Sphere::test(...) solves the quadratic for the two ray-sphere roots, and the nearest valid root inside [r.min_t, r.max_t] is kept. The full intersection routine computes the hit point r.o + t * r.d, uses the analytic normal (hit_p - center).unit(), and fills the same intersection fields as the triangle case.

    End-to-end flow:

    1. Sample a point inside the pixel in pathtracer.cpp.
    2. Generate a world-space ray in camera.cpp.
    3. Traverse the scene with pathtracer.cpp and bvh.cpp.
    4. Find the nearest primitive hit in triangle.cpp or sphere.cpp.
    5. Return the shading result to the pixel accumulator.
  2. Explain the triangle intersection algorithm you implemented in your own words.

    In triangle.cpp I used the Moller-Trumbore ray-triangle intersection test. The main idea is to solve for three unknowns at the same time: the ray time t and the barycentric coordinates b1 and b2. If those values exist and satisfy the triangle constraints, then the ray hits the triangle.

    I first build the two triangle edges e1 = p2 - p1 and e2 = p3 - p1. Then I compute s1 = cross(r.d, e2) and det = dot(s1, e1). That determinant tells me whether the ray is parallel to the triangle plane. If det is near zero, I reject the intersection because the solve is unstable.

    If the determinant is valid, I invert it and solve for the barycentric coordinates. With s = r.o - p1, I compute b1 = inv_det * dot(s1, s). If b1 falls outside [0, 1], the hit point is outside the triangle. I then compute s2 = cross(s, e1) and b2 = inv_det * dot(s2, r.d). If b2 < 0 or b1 + b2 > 1, that also means the point lies outside.

    Once the barycentric test passes, I compute t = inv_det * dot(s2, e2). That is the distance along the ray to the triangle. I only accept it if it lies within [r.min_t, r.max_t]; otherwise the hit is behind the ray origin, clipped out, or farther than a hit already found. On success I update r.max_t = t so later tests can ignore anything farther away.

    In the full intersect(...) routine, I recover the third barycentric weight as b0 = 1 - b1 - b2. I use (b0, b1, b2) to interpolate the vertex normals n1, n2, and n3, normalize the result, and store it in the Intersection along with t, primitive = this, and bsdf = get_bsdf(). So overall, the algorithm solves for the ray-plane hit, checks whether that point lies inside the triangle, and records it as the closest hit if it passes every test.

Part 2: Bounding Volume Hierarchy

  1. Walk through your BVH construction algorithm. Explain the heuristic you chose for picking the splitting point.

    The construction in bvh.cpp is a top-down recursive BVH build. For each recursive call, it takes a contiguous iterator range [start, end) over the primitive vector and does three things: it computes the bounding box of all primitives in that range, it computes a second bounding box over the centroids of those primitive boxes, and it creates a BVHNode for that range.

    If the range has <= max_leaf_size primitives, the recursion stops and the node becomes a leaf by storing start and end. That means leaf nodes do not copy primitives; they just refer to a subrange of the existing vector. If the range is larger than a leaf should be, the primitives are split into two groups and the builder recurses on both sides.

    The splitting heuristic I used is:

    1. Pick the axis with the largest centroid spread.
    2. Use the midpoint of the centroid bounds on that axis as the split position.
    3. Partition primitives by whether their own bounding-box centroid falls to the left or right of that split point.

    This works well because splitting along the widest centroid axis usually separates geometry better than using a fixed axis, centroid positions are cheap to compute, and a midpoint split is simple and fast enough for this assignment.

    There is one important degenerate case: if every centroid falls on one side of the split, the recursion would produce an empty child and the same full range again, which would cause infinite recursion. To avoid that, the implementation falls back to a median split with std::nth_element, forcing the two children to receive a balanced partition.

    So the overall strategy is: choose the widest centroid axis, split at the midpoint, and use a median fallback when the midpoint split degenerates.

Part 3: Direct Illumination

  1. Walk through both implementations of the direct lighting function.

    Common structure

    Both functions estimate the same integral:

    \[ L_o \approx \frac{1}{N}\sum \frac{f(\omega_o,\omega_i)\,L_i\,\cos\theta}{p(\omega_i)} \]

    They both:

    1. Build local shading coordinates from the normal using o2w and w2o.
    2. Compute hit_p and the local outgoing direction w_out.
    3. Evaluate the BSDF in local space with isect.bsdf->f(w_out, w_in).

    The relevant implementations are in pathtracer.cpp at estimate_direct_lighting_hemisphere and estimate_direct_lighting_importance.

    1. Uniform hemisphere sampling

    In the hemisphere version, the code uses num_samples = scene->lights.size() * ns_area_light and samples incoming directions uniformly over the hemisphere using hemisphereSampler->get_sample(). Because the sampling is uniform, the PDF is constant at 1 / (2pi).

    1. Sample a local incoming direction w_in on the hemisphere.
    2. Transform it to world space and cast a shadow ray from hit_p with min_t = EPS_F.
    3. If the shadow ray hits an emissive surface, use that emission as L_in.
    4. Accumulate f * L_in * abs_cos_theta(w_in) / pdf.
    5. Return the average over all samples.

    This estimator is unbiased, but it is noisy because most sampled directions do not actually point toward a light source.

    2. Light importance sampling

    In the importance-sampled version, the code loops over each light in scene->lights. It uses one sample for delta lights and ns_area_light samples for area lights, so the samples are targeted directly at the light sources instead of being spread uniformly across the hemisphere.

    1. For each light, call light->sample_L(hit_p, &w_in_world, &dist, &pdf).
    2. Reject invalid samples where pdf <= 0, dist <= EPS_F, or the local direction is backfacing.
    3. Cast a bounded shadow ray with min_t = EPS_F and max_t = dist - EPS_F to test occlusion.
    4. If the path is unoccluded, accumulate f * L_in * abs_cos_theta(w_in) / pdf.
    5. Average each light's contribution and sum across all lights.

    This approach is usually much less noisy, especially for area lights and point lights, because most samples contribute useful radiance. The switch between the two methods happens in pathtracer.cpp through direct_hemisphere_sample.

  4. Compare the results between uniform hemisphere sampling and lighting sampling in a one-paragraph analysis.

    With the same low sample budget, uniform hemisphere sampling is much noisier than light sampling because it spends most rays on directions that never hit a light, so soft shadows converge slowly and look grainy. In contrast, light sampling explicitly targets each light source, so nearly every sample contributes useful radiance, which produces cleaner penumbras, more stable brightness, and faster convergence even at modest -l values like 4 or 16. This difference is especially obvious in Cornell-box scenes with area lights, and it is even more important for delta or point lights, where uniform hemisphere sampling can be extremely inefficient while importance sampling remains reliable.

Part 4: Global Illumination

  1. Walk through your implementation of the indirect lighting function.

    Indirect lighting is handled in at_least_one_bounce_radiance as a recursive path-tracing estimator. The function first builds the local shading frame with o2w and w2o, then computes the hit point hit_p and the local outgoing direction w_out.

    1. Optionally add one_bounce_radiance at each bounce when accumulation is enabled, or only at the terminal bounce when it is disabled.
    2. Stop recursion when depth is exhausted, meaning r.depth <= 1.
    3. Apply Russian Roulette with continuation probability 0.7, while forcing the first indirect step to continue so GI is actually visible.
    4. Sample the BSDF with sample_f(w_out, &w_in, &pdf) and reject invalid samples.
    5. Transform w_in to world space, spawn the next ray from hit_p with min_t = EPS_F, decrement depth, and recurse if another surface is hit.

    The recursive contribution uses the Monte Carlo estimate:

    \[ f \cdot L_{\text{next}} \cdot \frac{\cos\theta}{\text{pdf} \cdot p_{\text{continue}}} \]

    This gets folded into the final pixel radiance by setting the camera ray depth in raytrace_pixel and then combining emitted light from zero_bounce_radiance with the recursive contribution in est_radiance_global_illumination. So the full estimator is emission plus direct and indirect bounces, with recursion controlled by depth and Russian Roulette.

Part 5: Adaptive Sampling

  1. Explain adaptive sampling. Walk through your implementation of adaptive sampling.

    Adaptive sampling means each pixel can receive a different number of rays depending on how noisy it still is, instead of always using the full fixed budget ns_aa. Easy pixels stop early, while difficult pixels keep sampling until they either converge or hit the maximum sample count. The convergence test uses a 95% confidence interval estimate:

    \[ I = 1.96 \sqrt{\sigma^2 / n} \]

    and stops when

    \[ I \le \text{maxTolerance} \cdot \mu \]

    Implementation walkthrough in pathtracer.cpp

    1. Set the maximum samples and batch cadence with num_samples = ns_aa and batch_size = max(1, samplesPerBatch).
    2. Handle the degenerate case: if num_samples <= 0, write black and store a sample count of 0.
    3. Track running pixel statistics using s1 = \sum x, s2 = \sum x^2, and samples_taken, where x is the per-sample illuminance.
    4. Sample rays normally by generating a camera ray, setting its depth, evaluating GI, and accumulating both radiance and statistics.
    5. Check convergence every batch by computing mu = s1 / n, sigma2 = (s2 - s1^2 / n) / (n - 1) with a nonnegative clamp, and then I = 1.96 * sqrt(sigma2 / n).
    6. If I <= maxTolerance * mu, stop early for that pixel.
    7. Write the final color using radiance / samples_taken and store samples_taken in sampleCountBuffer.

    That final step is what makes the _rate.png visualization meaningful, because the saved sample-count buffer directly reflects how many samples each pixel actually used.
